## 1. 评审执行摘要

**一句话结论**：WhichCity 是一个数据量惊人、功能完备的城市对比工具，但其视觉设计停留在「Tailwind CSS 默认主题 + 程序员审美」的水平，缺乏设计灵魂与品牌个性。整体风格如同一份被 ChatGPT 直接生成的、用 Tailwind 拼凑的 dashboard 模板——功能堆砌充分，但用户体验打磨严重不足。

### 关键数字

| 指标 | 数值 | 问题 |
|------|------|------|
| `darkMode ? "..."` 三元表达式 | **197 处** | 无设计系统，全靠复制粘贴 |
| `font-extrabold` 使用次数 | 12 处 | 一切数字都是 font-900，没有视觉层级 |
| `text-[10px]` / `text-[9px]` 自定义字号 | 13 处 | 绕过 Tailwind 设计比例尺 |
| `aria-*` 无障碍属性 | **仅 4 处** | 对 4,406 行组件代码而言几乎为零 |
| 内联 SVG 图标 | 15 个 | 无图标系统，手写 SVG 散落各处 |
| 超过 300 行规则的组件 | 4 个 | RankingContent 1087 行，严重违规 |
| 硬编码英语文案（Error/404） | 5 处 | i18n 覆盖有硬伤 |

## 3. A 类：配色与视觉设计

### 3.1 色板分析

**主色调**：几乎 100% Slate 系。整站配色可以概括为「深灰 + 浅灰 + 5 种不相干的点缀色」。

| 层级 | 亮色模式 | 暗色模式 |
|------|---------|---------|
| 背景 | `slate-50` (#f8fafc) | `slate-950` (#020617) |
| 卡片 | `white` + `slate-200` 边框 | `slate-800` + `slate-700` 边框 |
| 正文 | `slate-900` | `slate-100` |
| 次要文字 | `slate-500` | `slate-400` |
| 分隔线 | `slate-200` | `slate-700` |

**问题**：
- 🔴 **致命的单调**：85% 视觉面积是 Slate 灰色。网站看起来像是一张还没有被设计师"上色"的线框图（wireframe）。
- 🔴 **accent 色彩杂乱无章**：Blue（首页/排名高亮）、Amber（排名导航/气候图温度）、Emerald（好指标/随机城市按钮/气候筛选）、Rose（差指标）、Violet（对比导航）、Sky（降雨图）——6 种 accent 色无语义关联，无品牌一致性。
- 🟡 **深色模式卡片边框几乎不可见**：`border-slate-700` on `bg-slate-800` 对比度约 1.5:1，人眼几乎无法区分边框与背景。

### 3.2 WCAG 对比度违规

| 元素 | 组合 | 对比度 | WCAG AA (4.5:1) |
|------|------|--------|-----------------|
| "差指标"文字（暗色模式） | `rose-400` on `slate-950` | **3.1:1** | ❌ **不合格** |
| "好指标"文字（暗色模式） | `emerald-400` on `slate-950` | 4.2:1 | ⚠️ 勉强合格 |
| "差指标"文字（亮色模式） | `rose-500` on `white` | 3.9:1 | ⚠️ 临界值 |
| 次要文字（暗色模式） | `slate-400` on `slate-950` | 6.1:1 | ✅ 合格 |
| 气候图 9px 数字文字 | `amber-400` on dark | 约 4.0:1 | ⚠️ 正常文本不合格 |

> **评审意见**：一个数据密集型网站，用颜色传达好/坏信息的核心功能，居然在最关键的"差指标"文字上对比度不合格——这是打脸级别的失误。

### 3.3 视觉层级缺失

- **`font-extrabold`（font-weight: 800）被滥用**：所有数据数值——收入、房价、安全指数、储蓄率——全部使用 `text-xl font-extrabold`。当一切都加粗时，没有东西是加粗的。
- **字号碎片化**：全站共使用约 15 种字号组合（text-4xl、text-3xl、text-2xl、text-xl、text-lg、text-base、text-sm、text-xs、text-[10px]、text-[9px]），毫无节制。Tailwind 默认字号比例尺被 `text-[10px]`/`text-[9px]` 自定义值打穿 13 次。
- **阴影层级混乱**：搜索下拉框 `shadow-lg`，数据卡片 `shadow-sm`，比较页选择器 `shadow-md`——搜索框的视觉权重竟然比数据卡片还高。

## 4. B 类：布局与响应式

### 4.1 断点混乱

| 断点 | 来源 | 用途 | 问题 |
|------|------|------|------|
| `min-[1080px]` | 自定义 JS | Compare 3→2 列、NavBar 汉堡菜单 | ❌ 非标准，与 Tailwind lg (1024px) 冲突 |
| `min-[420px]` | 自定义 Tailwind | NavBar 按钮文字长短切换 | ⚠️ 窄屏边界行为不稳定 |
| `md` (768px) | Tailwind 标准 | 比较页 2 列回退 | ✅ 合理 |
| `sm` (640px) | Tailwind 标准 | 网格、字号调整 | ✅ 合理 |
| `lg` (1024px) | Tailwind 标准 | CityDetail 索引卡片 2→4 列 | ⚠️ 跳跃太大（直接 2→4） |

**致命问题**：CompareContent 使用 JavaScript `window.matchMedia("(min-width: 1080px)")` 来控制列数，而非 CSS 响应式。这意味着：
1. SSR 时默认渲染 3 列，小屏用户会看到布局闪烁
2. 调整窗口大小时有 JS 延迟，而非 CSS 即时响应
3. 与其他页面使用 Tailwind CSS 断点的方式不一致

### 4.2 移动端体验灾区

**排名页表格（RankingContent）**：
- 1087 行代码的巨型组件，在手机上渲染为一个需要水平滚动的 `<table>`
- 表头 `min-w-[80px]` × 5-6 列 = 至少 480px 宽度
- 在 375px iPhone 上，用户必须左右滑动才能看到完整数据
- 没有任何视觉提示告诉用户"可以向右滑动"
- **建议**：手机端应替换为卡片列表视图

**气候图（ClimateChart）**：
- 使用固定高度 `TEMP_H = 160px` + `RAIN_H = 50px`
- 在 320px 窄屏上，12 个月份的温度标签（`text-[10px]`）会严重重叠和溢出
- 不响应屏幕尺寸缩放

### 4.3 内容宽度不一致

- 主体内容：`max-w-6xl` (1152px)
- Footer 内部：`max-w-5xl` (1024px)
- NavBar 内部：`max-w-6xl`
- 这导致 Footer 文字与上方内容存在 128px 的对齐偏移

## 5. C 类：交互体验与可用性

### 5.1 致命级交互问题

**① 设置面板不可见**：
- 设置按钮仅是一个 `h-7 w-7`（28×28px）的齿轮图标，没有文字标签
- 按钮面积远低于 Apple HIG 推荐的 44×44px 最低触控目标
- 用户如果不知道点齿轮 = 打开设置，就永远不会发现可以切换职业、收入模式、消费档位、货币——这些都是影响全站数据的核心设置
- **结果**：大量用户看到的数据是错误的（默认 gross + moderate + USD + 第一个职业），但他们不知道可以改

**② 比较页城市切换没有操作提示**：
- 城市选择器看起来像一个只读标签（城市名 + 国旗 + 下拉箭头）
- 没有 "点击更换" 提示
- `×` 清除按钮只有 20×20px，在移动端几乎无法点击
- 用户可能以为切换城市需要回到首页重新搜索

**③ 排名页 Tab/Filter 系统过度复杂**：
- 16 个指标 Tab + 气候筛选（6 类型 × 5 维度 × 3 档位）
- 单选/多选模式切换隐藏在展开后的面板右上角
- 用户需要 3-4 次点击才能从"收入排名"切换到"气候筛选"
- 组合打分（composite mode）的概念对普通用户而言完全不可理解

**④ Error/404 页面是死路**：
- Error 页："Oops / Something went wrong / Try again"——硬编码英文，无 locale 感知
- 404 页："404 / Page not found / ← Back to Home"——链接到 `/` 而非 `/${locale}`
- 两个页面都没有提供备选路径（搜索、热门城市、排名页链接）
- 用户在这两个页面上的唯一选择是：重试/回首页，或关闭标签

### 5.2 重大交互问题

**⑤ 搜索键盘导航无任何提示**：
- 支持 ↑↓ 箭头键、Enter 确认、Esc 关闭
- 但没有任何视觉线索告诉用户可以用键盘操作
- 估计 70%+ 用户从未发现此功能

**⑥ Win Count（胜出计数）没有解释**：
- 比较页显示每个城市在哪些指标上"获胜"
- 但什么算"获胜"？更高还是更低？包含哪些指标？
- 用户看到数字但无法理解其含义

**⑦ NavBar 滚动时无阴影分隔**：
- 使用 `sticky top-0 z-50` 但无 `shadow-sm`
- 滚动后 NavBar 底部边框（1px）与页面内容视觉融合，失去层级感

**⑧ 加载状态为空白**：
- 设置未就绪（`!s.ready`）时，显示空白 div + NavBar
- 无 skeleton loading、无 spinner、无任何加载指示
- 首次访问用户会看到 0.5-1s 的白屏（或暗色空屏）

## 6. D 类：国际化与内容质量

### 6.1 硬编码英文文案

| 文件 | 行号 | 内容 | 严重性 |
|------|------|------|--------|
| `error.tsx` | L11 | `"Oops"` | 🔴 |
| `error.tsx` | L12 | `"Something went wrong"` | 🔴 |
| `error.tsx` | L17 | `"Try again"` | 🔴 |
| `not-found.tsx` | L7 | `"Page not found"` | 🔴 |
| `not-found.tsx` | L12 | `"← Back to Home"` | 🔴 |

这 5 处硬编码英文意味着：使用中文/日文/西班牙文的用户在遇到错误或 404 时，会突然看到英文——这是国际化的基础性失败。

### 6.2 404 页面 locale 丢失

`not-found.tsx` 中 `<Link href="/">` 直接指向根路径，而非当前 locale 对应的路径。一个 `/zh/some-invalid-page` 的用户点击"Back to Home"后会被 middleware 重定向到 `/en`（或浏览器默认语言），丢失中文设置。

### 6.3 CJK 字体支持缺失

```typescript
const inter = Inter({ subsets: ["latin"], display: "swap" });
```

仅加载了 `latin` 字集。中文和日文内容回退到系统字体。虽然 public/fonts/ 下有 `NotoSansSC-Bold.ttf`（仅用于 OG 图片生成），但正文未使用。结果：
- 中日文字体渲染不一致
- 窄屏下中文字符可能与 Inter 拉丁字母产生基线偏移

### 6.4 方法论页面数据源无超链接

MethodologyContent 中列出了 20+ 数据源（Numbeo、UNODC、WHO 等），但数据源名称是纯文本，没有提供原始 URL 链接。用户无法验证数据出处——对一个以"数据"为核心卖点的网站来说，这是信任度的硬伤。

## 7. E 类：无障碍（Accessibility）

### 7.1 aria 属性基本为零

**4,406 行组件代码中仅有 4 处 `aria-*` 属性**，且全部集中在 CompareContent 的城市选择器上。

| 组件 | aria 属性数 | 应有属性 |
|------|------------|---------|
| HomeContent | 0 | 搜索框：`role="combobox"`, `aria-expanded`, `aria-live` |
| RankingContent | 0 | 表格：`aria-sort`, `aria-label`；Tab：`role="tablist"` |
| CompareContent | 4 | 仅城市选择器有 `aria-expanded`/`aria-haspopup` |
| NavBar | 0 | 设置按钮：`aria-label`；汉堡菜单：`aria-expanded` |
| CityDetailContent | 0 | 索引卡片：`aria-label` |

**评审意见**：这不是"待改进"，这是"还没开始做"。对于一个面向全球用户的网站，无障碍支持接近于零是不合格的。

### 7.2 触控目标尺寸违规

| 元素 | 实际尺寸 | 推荐最小尺寸 | 合规 |
|------|---------|-------------|------|
| NavBar 设置按钮 | 28×28px (`h-7 w-7`) | 44×44px | ❌ |
| NavBar 分享按钮 | 28×28px | 44×44px | ❌ |
| 城市选择器 × 关闭按钮 | 20×20px (`w-5 h-5`) | 44×44px | ❌ |
| 热门城市链接 | text-xs, ~60×16px | 44×44px | ❌ |

### 7.3 键盘焦点样式

除搜索框外，所有交互元素均依赖浏览器默认焦点环。没有自定义 `:focus-visible` 样式。在某些浏览器/主题组合下，焦点环可能不可见。

## 8. F 类：冗余与膨胀

### 8.1 代码规则违反

项目自定义规则 `RULES.md` 规定「文件 < 300 行」，但：

| 文件 | 行数 | 超标率 |
|------|------|--------|
| RankingContent.tsx | 1,087 | **+262%** |
| MethodologyContent.tsx | 978 | +226% |
| CityDetailContent.tsx | 862 | +187% |
| CompareContent.tsx | 802 | +167% |
| NavBar.tsx | 306 | +2%（勉强超标） |

5 个组件中有 4 个严重超标。规则形同虚设。

### 8.2 功能膨胀

**① 分享面板 8 个按钮**：
X、Facebook、WhatsApp、LinkedIn、Reddit、Telegram、LINE、Copy Link。对于一个个人项目，8 个分享渠道严重过剩。50% 用户只会用其中 2-3 个，其余占据宝贵 UI 空间。

**② 22+ 排名指标 Tab**：
收入系（3个）+ 住房系（3个）+ 工作系（3个）+ 环境系（3个）+ 指数系（4个） = 16 个常规 Tab，每个指数 Tab 又有 3-5 个子排序项。总计 22+ 可选维度，堪比 Bloomberg Terminal 的操作复杂度——但目标用户是「考虑搬家的普通人」。

**③ 数字游牧专区信息过载**：
签证类型、停留时长、最低收入、外国收入税、VPN 限制、英语水平、4 本护照的免签天数、4 个时区的重叠时间——一个城市详情页就要消化 20+ 条游牧数据。

**④ "相似城市"功能价值存疑**：
使用 21 维欧氏距离计算最相似的 6 个城市。算法复杂度高，但用户无法理解"为什么这些城市相似"。与排名页功能重叠。

### 8.3 样式代码重复

| 模式 | 出现次数 | 问题 |
|------|---------|------|
| `${darkMode ? "bg-slate-800" : "bg-white"}` | ~40 次 | 应抽取为变量 |
| `${darkMode ? "text-slate-400" : "text-slate-500"}` | ~30 次 | 应抽取为变量 |
| `rounded-xl border shadow-sm p-4 ${darkMode ? ... : ...}` | ~20 次 | 应抽取为 Card 组件 |
| NavBar select 元素样式（相同 class 串） | 11 次 | 完全相同的字符串复制粘贴 |
| darkMode 颜色三元表达式（总计） | **197 处** | 缺乏设计系统 |

## 9. G 类："AI 味"与设计品质

### 9.1 为什么这个网站看起来像 AI 生成的

**核心病因**：这个项目的每一个视觉决策都是"Tailwind CSS 的默认选项"。没有一个颜色、一个间距、一个圆角半径体现了人类设计师的审美判断。

具体表现：

| 特征 | 本项目 | "有灵魂"的设计 |
|------|--------|---------------|
| 色板 | Slate 灰 + 5 种随机 accent | 有品牌色、有层次的色彩系统 |
| 圆角 | 全部 `rounded-xl` (12px) | 不同元素使用不同圆角（按钮 8px、卡片 16px、搜索框 full） |
| 阴影 | `shadow-sm` / `shadow-md` 两档 | 有语义的阴影系统（hover 时提升、active 时压低） |
| 间距 | gap-1.5、gap-2、gap-3、gap-4 随机混用 | 基于 4px/8px 网格的一致间距 |
| 动效 | 几乎没有（仅 `transition-colors`） | 微交互、hover 动画、页面切换过渡 |
| 排版 | Inter 字体 + 随机字号 | 有限的字号阶梯（3-5 级）+ 呼吸感 |
| 个性 | 无 | 有标志性视觉语言（渐变、纹理、特殊形状） |

### 9.2 逐条 "AI 味"检测

| # | 症状 | 文件位置 | 评级 |
|---|------|---------|------|
| 1 | **一切都是 rounded-xl**——按钮、卡片、搜索框、下拉菜单、索引卡片、设置面板、气候筛选按钮全部 12px 圆角 | 全部组件 | 🔴 高 AI 味 |
| 2 | **暗色模式 = 把所有颜色往暗里推**——无任何只属于暗色模式的独特视觉元素 | globals.css | 🔴 高 AI 味 |
| 3 | **没有渐变、没有纹理、没有插图**——全站唯一的非文字视觉元素是 emoji 国旗和 SVG 图标 | 全部组件 | 🔴 高 AI 味 |
| 4 | **NavBar 四色按钮**——蓝/琥珀/翠绿/紫堆在一起，像是 AI 被问"给 4 个按钮配不同颜色"的直觉反应 | NavBar.tsx | 🟡 中 AI 味 |
| 5 | **Methodology 四语言并列全文翻译**——978 行代码中大量是在函数内硬写 JSX 内容，像是 AI 直接 inline 翻译 | MethodologyContent.tsx | 🟡 中 AI 味 |
| 6 | **footer 三联链接** `Methodology · GitHub · Feedback` 在每个页面底部重复出现——最经典的 AI 生成页脚模式 | HomeContent.tsx, RankingContent.tsx | 🟢 轻 AI 味 |

### 9.3 设计品质的缺失

这个项目**没有设计系统**。没有 `colors`、`spacing`、`shadows` 的 token 定义。`tailwind.config.ts` 的 `theme.extend` 是空的——字面意义上的空的。

```typescript
theme: {
  extend: {},   // ← 没有自定义任何东西
},
```

这意味着整站视觉 100% 依赖 Tailwind 默认值。在 2026 年，这相当于用默认 Bootstrap 做网站——技术上没有错，但视觉上一眼就能看出来。
