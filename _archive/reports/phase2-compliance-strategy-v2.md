## 1. 五轮讨论纪要

### 第一轮：创始人原则意味着什么

**📋 PM**：

先把话说明白。创始人要求的不是"修修补补的合规"——是**彻底清洗**。包括已经在手里的 Numbeo 数据也要扔掉。这意味着：

- costModerate / costBudget / monthlyRent / housePrice → 全部归零
- numbeoSafetyIndex → 归零
- Numbeo 城市薪资基准线 → 归零

我查了薪资系统的依赖关系：当前 3,600 个薪资数据点中，**87.2% 依赖 Numbeo 作为城市级薪资锚点**。129 个非美国城市的全部薪资计算都以 Numbeo 的城市平均税后月薪为起点。清洗 Numbeo 不仅清掉了成本数据——还清掉了薪资系统的地基。

说实话，这是一次产品级别的"推倒重建"。

**⚖️ Legal**：

从法律角度说，创始人的态度是我见过最干脆的。没有灰色地带、没有"先用着再说"、没有侥幸心理。这在法务实务中非常罕见——大多数创始人都想在"合规"和"保留数据"之间找平衡。这位创始人选了最难走但最干净的路。

坦率说，如果 Numbeo 的数据确实是脚本爬的，那按照 Numbeo TOS 原文（"Automated data collection methods... are **strictly prohibited**"），这些数据从获取方式上就有瑕疵。创始人选择全部清除是**道义上正确的**。

但是——我需要提醒：全面清洗后，MEDIUM 风险的数据源（RSF、Freedom House、MIPEX、WJP）怎么处理？这些不是"偷"来的，是公开下载的。但许可条款不明确。

**📈 Marketing**：

我支持清洗，但我需要说一句不好听的：**清洗后的产品在短期内会失去大约 60% 的用户可感知价值。** 

用户来城市对比工具干什么？看"这个城市月消费多少、租金多少"。清掉这些数据后，产品就变成了"告诉你税后能赚多少，但不告诉你要花多少"。这在产品体验上是不完整的。

但——创始人说了"我不怕麻烦，我有的是时间"。如果他真的愿意投入时间从政府来源重建成本数据，那最终结果反而是：**世界上唯一一个数据来源 100% 透明可查的城市对比工具。** 这个叙事比"我们有 150 个城市的数据"强得多。

**💰 Investor**：

我看到的是创始人的一个核心判断：**他不愿意把产品的生命线交到别人手里。**

这不是在说数据合规——这是在说**产品独立性**。如果你的核心数据来自 Numbeo，那 Numbeo 就是你的供应商。供应商可以涨价、断供、被收购。创始人拒绝这个依赖关系本身。

从投资角度，这是一个**短期极其痛苦、长期极其正确**的决策。因为：
1. 如果做成了（自有数据体系），护城河反而更深
2. 税制引擎完全不受影响——这是真正的 IP
3. "站着赚干净钱"的品牌调性，在 2026 年的数据伦理大环境下是加分项

**但我必须问一个关键问题**：清洗之后、重建完成之前——产品能不能活着？能不能有一个"最小可用状态"让用户至少觉得有价值？如果中间有半年是半成品，用户会流失。

我做了一个完整的盘点。清洗后产品分三层：

**⚖️ Legal**：

MEDIUM 风险项需要逐个判断。我的建议：

1. **RSF Press Freedom Index**：数据公开免费下载，RSF 是新闻自由倡导组织，TOS 页面 404 不等于"禁止使用"。RSF 的使命就是让这些数据被广泛使用。**保留，但加 attribution，同时发一封简单询问邮件**——这不是"商业谈判"，是尽职调查。

2. **Freedom House**：TOS 原文明确说"Use for noncommercial purposes is permitted"。WhichCity 目前就是非商业的。当产品商业化时再发邮件申请。现阶段**可保留，加 attribution**。

3. **WJP Rule of Law**：数据公开下载，条款不明。与 RSF 类似——一个致力于法治推广的非营利组织，不太可能阻止数据被引用。**保留，加 attribution，发询问邮件**。

4. **MIPEX**：EU Horizon 2020 资助项目，按 EU 的 Open Research Data 政策，公共资助的研究数据原则上应该开放获取。但这只是"原则上"。**保留，加 attribution，发询问邮件**。

5. **Georgetown WPS**：学术机构发布的研究数据，使用单个国家级聚合分数。学术惯例允许引用。**保留，加 attribution**。

**关键区分**："发一封询问邮件确认数据使用条款"≠"谈商业合作"。前者是数据使用者的尽职调查责任，后者是商业行为。创始人拒绝的是后者，前者反而是"干净透明"原则的体现。

**📈 Marketing**：

同意法律顾问的区分。补充一点：如果我们给 RSF / Freedom House / WJP / MIPEX 发邮件，应该明确说"我们是一个开源的、非营利的城市比较工具"——这和说"我们是一个想挂广告赚钱的商业网站"得到的回复会完全不同。

**💰 Investor**：

回到我上一轮的问题——清洗后、重建前的"最小可用产品"是什么？

我看到 PM 的盘点后，我认为：

**最小可用产品 = 税制引擎 + 美国 21 城薪资对比 + 安全/医疗/治理指数 + 气候/空气/航线**

这个组合即使没有成本数据也有独立价值。用户可以问："作为软件工程师，在纽约和旧金山之间选，税后收入差多少？安全和医疗怎么样？"——这个问题完全可以回答。

"月消费多少"暂时回答不了，但坦率地说，这是 Numbeo 能回答而不是 WhichCity 能回答的——如果用户想看消费，他们本来就会去 Numbeo。WhichCity 的独特价值不在消费数据，而在税制引擎。

### 第三轮：薪资系统重建——核心争论

**📋 PM**：

薪资系统是最复杂的重建任务。让我说清楚当前的依赖链：

```
BLS OEWS (21 美国城市)              → 直接使用 ✅
Numbeo 城市税后月薪 (148 城市)       → 作为 net → gross 锚点 🔴
各国政府统计局职业比率 (80 国)       → 比率系数 ✅
ILO ISCO 大组比率                   → 回退比率 ✅
```

移除 Numbeo 后，129 个非美城市的薪资计算少了**城市级锚点**。国家级替代品有：

- **ILO ILOSTAT 平均工资** (CC BY 4.0, ~100 国)：国家级，但无法区分同国不同城市
- **各国统计局**：Quality A 的 10 国（日/加/英/德/法/新加坡/瑞/韩/捷/新西兰）有城市或地区级工资数据

我的方案是**三阶段重建**：

**第一阶段（立即）：降级但可用**
- 美国 21 城：保持现状（BLS 直接值）
- Quality A 10 国城市（~25-30 城市）：从政府来源获取城市/地区级工资，作为新锚点
- 其余城市：用 ILO 国家平均工资作为锚点，放弃城市间差异，置信度标记下降

**第二阶段（1-3 个月）：逐国重建**
- Quality B 22 国：逐个研究政府统计局数据，寻找城市/地区级工资
- 每完成一个国家，该国城市的薪资精度就恢复

**第三阶段（持续）：独立数据体系**
- 建立 WhichCity 自己的"城市工资指数"，完全基于政府来源
- 记录每个数据点的来源 URL，任何人可以验证

**⚖️ Legal**：

从法律角度补充：ILO ILOSTAT 是 CC BY 4.0，World Bank 是 CC BY 4.0，各国政府统计局的公开发布数据在绝大多数国家属于公共信息或开放许可。这条重建路径**从头到尾合法合规**。

唯一需要注意的是：部分亚洲国家的统计局网站有 TOS 限制（如中国国家统计局声明"版权所有"）。但统计数据属于**事实数据（factual data）**，在大多数法律体系中不受著作权保护——你不能对"东京平均月薪 38 万日元"这个事实主张版权。只是不能复制整个数据库的结构和呈现方式。

**📈 Marketing**：

第一阶段意味着什么？

- 21 美国城市：精确
- ~25-30 发达国家城市：精确
- 其余 ~65 城市：国家级精度（同国城市显示相同薪资基准）

这其实没那么糟。对于用户来说：

> "旧金山的软件工程师税后年收入 $118K（BLS 直接数据），东京的同一职业税后年收入 ¥6,800K（厚劳省赁金構造基本統計調査 × 税制引擎）"

这两个数字都是有来源的、可验证的。用户不会在意"东京的数据是国家级 vs 城市级"——他们在意的是"这个数字可信吗"。

而置信度系统本来就已经建好了。把精度差异诚实地告诉用户反而是加分项。

**💰 Investor**：

我唯一的担心是**时间窗口**。

第一阶段覆盖 ~50 城市的精确薪资，加上其余城市的国家级估算——这个产品状态能不能上市（哪怕是 soft launch）？

如果能，那重建过程就是一个"持续改进"的过程，不需要等到 100% 完成才发布。如果不能，那就有一个"不可用期"，这对刚起步的产品是致命的。

我的判断：**可以上市。** 50 个城市的精确薪资 + 120 城市的国家级估算 + 88 国税制引擎 + 安全/医疗指数——这已经比市面上任何免费工具都强了。差的只是 Numbeo 级别的消费数据。

### 第四轮：台北问题、市场策略、产品定位

**📋 PM**：

先处理台北问题。

创始人明确表态：视台湾为独立国家，不考虑中国大陆市场。问题是：保留还是移除台北？

我先列事实：
- 台北 (ID 61) 在数据层面，country = "台湾"
- 税制引擎中台湾有独立税制（与中国大陆不同的累进制）
- 部分国际组织数据缺失（WB 中台湾很多指标为 null，homicideRate 等 5 项缺失）
- 台湾用户是 zh 语言的重要受众

**⚖️ Legal**：

法律角度：在网站上把台湾标注为独立国家/地区，在中国大陆法律下是违法的（《反分裂国家法》），在中国大陆运营的平台不可这样做。但 WhichCity 的域名（.run）、服务器（假设不在中国大陆）、创始人（假设在中国大陆以外运营）——如果以上成立，那么中国大陆法律的域外管辖力有限。

在全球其他主要市场（美/欧/日/台/拉美），将台湾的税制和生活数据独立列出是**标准做法**。NomadList、Teleport、Numbeo 都这么做。

**我的建议：保留台北，但做两件事**：
1. 确认数据层面 country 字段使用"台湾 (Taiwan)"而非涉及主权声明的措辞
2. 在网站不做任何政治声明——数据本身就是事实陈述（台湾有独立税制、独立货币、独立签证政策——这些都是事实）

**📈 Marketing**：

**强烈建议保留台北。**

理由：
1. 台湾有 2,300 万人口，高度数字化，是高质量用户群
2. 日本市场和台湾市场高度关联——很多台湾人对日本生活感兴趣，反之亦然
3. 移除台北反而会被视为"自我审查"——在目标市场（日/台/英语圈）里这是减分项
4. 创始人说"站着赚钱"——如果连自己的信念都要藏着，那就不是站着

**市场策略调整**：

| 原计划 | 调整后 |
|--------|--------|
| #1 中国出海人群 | ❌ 移除 |
| #2 全球 tech workers | ✅ **提升为 #1** |
| #3 日本脱出组 | ✅ **提升为 #2** |
| #4 拉美外迁人群 | ✅ **保持 #3** |
| — | **新增 #4: 台湾出走族**（考虑移居日/新/澳/加的台湾科技人） |

放弃大陆市场后，zh locale 的定位从"面向中国大陆 + 台湾 + 海外华人"变为"面向台湾 + 港澳 + 海外华人"。内容完全不需要调整——繁简体差异在搜索引擎上的影响有限，台湾用户可以阅读简体中文。

**💰 Investor**：

同意保留台北。从投资视角看，一个有明确价值观的产品比一个讨好所有人的产品更有品牌辨识度。

但我要说一句逆耳的话：**不要在产品上做任何政治声明**。台湾作为一个独立的税务/签证/货币管辖区出现在产品中——这是事实；但如果在 About 页面写"台湾是一个独立国家"——那就是政治声明。

产品应该让数据说话，不需要创始人的政治观点出现在任何公开页面上。

**📋 PM**：

四方共识：**保留台北，不做政治声明，让数据本身成为陈述**。

接下来谈产品定位偏移。创始人说可以接受偏移但不能偏出"whichcity.run"的精神。什么是"whichcity"的精神？

> "Which city should I move to?"

这个问题的本质是**帮人做移居决策**。做决策需要什么？
1. 我在那里能**赚多少**？（税后收入）← 有
2. 我在那里要**花多少**？（生活成本）← 暂时没有，需要重建
3. 那里**安全吗**？（安全指数）← 有，需要重设计
4. 那里**医疗好吗**？（医疗指数）← 有
5. 那里**自由吗**？（治理/自由指数）← 有，需要精简
6. 那里**气候/环境怎样**？← 有

6 个维度中，只有"花多少"暂时缺失。产品定位不需要偏移——只需要明确**阶段性**：

> Phase A（现在）：WhichCity 告诉你在全球 120 个城市能赚多少（税后）以及那里安不安全、医疗好不好
> Phase B（重建后）：WhichCity 告诉你在全球 120 个城市的完整生活图景——赚多少、花多少、存多少

**这不是定位偏移，是定位的分阶段兑现。**

### 第五轮：执行路径、时间线、营收

**📋 PM**：

关键路径：

```
第 1 周: 数据清洗 + 零成本替代
  ├── 删除所有 Numbeo 数据（cost/rent/housePrice/numbeoSafety/salary baseline）
  ├── 替换 5 个字段（UNODC→WB, IEP→WB, Gallup→WB, EIU→V-Dem, Ookla→M-Lab）
  ├── 安全指数重设计（全 WB 子指标）
  ├── 薪资系统第一阶段（ILO 国家级锚点 + Quality A 国家城市级数据）
  └── 治理/自由指数精简
```

## 4. 生活成本数据重建方案

### 4.1 核心策略：从政府来源逐国重建

**不走捷径。不估算。不"参考"任何受限来源。一个国家一个国家地从官方统计局获取。**

### 4.2 数据组件

生活成本 = 食品 + 住房/租金 + 交通 + 通信 + 基本消费

与其像 Numbeo 那样给出单一"月消费"数字，WhichCity 可以做得更好——给出**组件化的成本分解**：

```
新 costModerate 计算:
  = 食品支出 (政府 CPI 食品篮 × 当地价格)
  + 住房租金 (政府租金数据 / 住房调查)
  + 交通支出 (公交月票 + 平均通勤成本)
  + 通信 (手机+网络套餐均价)
  + 基本消费 (服装/日用/娱乐 — CPI 组件)
```

## 7. 台北与市场策略裁决

### 7.1 台北裁决：保留

**四方一致同意保留台北。**

理由：
1. **价值观一致**：创始人明确立场，移除台北等于自我审查，违背"站着"的原则
2. **数据事实**：台湾有独立税制（所得税法）、独立货币（TWD）、独立签证政策。作为城市对比工具列出台北是**陈述事实**，不是政治声明
3. **市场逻辑**：放弃大陆市场后，台湾（2,300 万人口、高数字化率）是 zh locale 的核心受众
4. **产品完整性**：台北是亚洲重要城市，移除会导致产品在东亚覆盖上出现明显缺口
5. **竞品对照**：Numbeo/NomadList/Teleport 均将台湾独立列出

**执行要点**：
- country 字段保持 "台湾" / i18n: "Taiwan"
- 不在产品任何页面做政治声明
- 税制引擎已有台湾完整税制，不需修改
- 台北的部分 WB 数据缺失（台湾不是 WB 成员）→ 使用台湾政府公开数据补齐（DGBAS/主計總處）

## 8. 清洗后的产品定位

### 8.1 定位偏移评估

创始人说："能接受偏移，但不能偏出 whichcity.run 的精神。"

**"whichcity" 的精神 = "Which city should I move to?"**

清洗后的产品能不能回答这个问题？

| 决策维度 | 清洗前 | 清洗后 (立即) | 清洗后 (重建完) |
|---------|--------|-------------|---------------|
| 税后收入 | ✅ 精确 | ✅ 精确（美+A国）/ 🟡 估算（其余） | ✅ 精确 |
| 生活成本 | ✅ 有 | ❌ 暂无 | ✅ 重建（政府来源） |
| 安全 | ✅ 有 | ✅ 有（WB 重设计） | ✅ 有 |
| 医疗 | ✅ 有 | ✅ 有 | ✅ 有 |
| 治理/自由 | ✅ 有 | ✅ 有 | ✅ 有 |
| 气候 | ✅ 有 | ✅ 有 | ✅ 有 |

**结论：不需要定位偏移。** "Which city should I move to?" 这个问题的回答暂时不完整（缺成本数据），但核心功能（税后收入 + 安全 + 医疗）足以让用户获得有价值的决策信息。

### 8.2 阶段性定位

**Phase A（清洗完成后，第 3 周）**:

> **WhichCity — 全球移居税务引擎**
>
> 选择你的职业，对比 120 座城市的税后收入。
> 88 国税制引擎 · 25 种职业 · 安全/医疗/治理指数 · 100% 开放数据

强调"税后收入对比"和"开放数据"。不提成本数据。

**Phase B（成本数据上线后，第 12-20 周）**:

> **WhichCity — 全球移居决策引擎**
>
> 选择职业和身份，用数据找到更好的城市。
> 税后收入 · 生活成本 · 安全 · 医疗 · 气候 · 100% 政府来源数据

恢复原始定位，但加上"100% 政府来源数据"作为品牌标签。

### 8.3 这不是"降级"，这是"升级路径"

从 Numbeo 数据到政府来源数据，是**数据质量的升级**——只不过升级需要时间。

在这个时间窗口里，产品以税制引擎为核心叙事，是一个**完全诚实的定位**。不夸大、不虚报、不用别人的数据装门面。

## 9. 透明化品牌战略

### 9.1 创始人说的"公开可查"怎么实现

**三层透明机制**：

#### 第一层：DATA_PROVENANCE.md（数据溯源文件）

在代码仓库中维护一份每个字段的溯源文件：

```markdown
## costModerate
- 状态: 🔴 暂无数据（重建中）
- 原来源: Numbeo（已清除 — TOS 禁止抓取和商用）
- 重建计划: 从各国统计局政府来源逐国重建
- 预计时间: 2026 Q3

## safetyIndex
- 子指标 1: Intentional Homicide Rate (30%)
  - 来源: World Bank API — VC.IHR.PSRC.P5
  - 许可: CC BY 4.0
  - 最后更新: 2026-04
  - 验证 URL: https://data.worldbank.org/indicator/VC.IHR.PSRC.P5
- 子指标 2: Political Stability (25%)
  - ...
```

#### 第二层：Methodology 页面

网站上的 Methodology 页面扩展为完整的数据治理中心：
- 每个指数的计算公式（公开）
- 每个数据字段的来源和许可
- 数据更新时间线
- 已知的数据限制和精度范围
- "为什么我们不用 Numbeo" 的简短说明

#### 第三层：开源数据管道

如果创始人愿意，数据采集和计算脚本也可以开源。这意味着：
- 任何人可以复现 WhichCity 的数据
- 任何人可以验证任何数字
- Bug 可以被社区发现

**这三层透明度在城市对比工具领域是前所未有的。** 它直接回应了创始人"干净、公开、透明"的要求。

### 9.2 "站着赚钱"的品牌叙事

当竞品说"我们有最全的数据"时，WhichCity 说：

> "我们的每一个数字都有来源链接。你不需要信任我们——你可以自己验证。"

这在 2026 年的互联网环境中是**强有力的差异化**。AI 生成内容泛滥、数据来源不透明、隐私丑闻频发——一个"100% 可验证"的工具本身就是品牌。

## 10. 营收路径

### 10.1 不做什么

| 变现方式 | 是否接受 | 理由 |
|---------|---------|------|
| Google AdSense | ❌ 不推荐 | 广告内容不可控，可能与"干净"形象冲突 |
| 向数据源付费 | ❌ 创始人明确拒绝 | 原则 #2 |
| 卖用户数据 | ❌ | 与透明原则根本冲突 |

### 10.2 推荐的变现路径

**阶段 1：不变现（0-6 月）**

专注产品质量和数据重建。创始人说"有的是时间"——这段时间投在产品上而不是变现上。

**阶段 2：适度变现（6-12 月）**

| 方式 | 说明 | 与"干净"原则是否一致 |
|------|------|---------------------|
| **赞助** | 底部一行 "Supported by [sponsor]"。只接受与移居相关的正当企业（国际搬家/保险/银行）| ✅ 透明、选择性合作 |
| **高级功能** | PDF 报告导出、10 城市以上对比、历史趋势图 | ✅ 用户付费购买增值服务 |
| **GitHub Sponsors / Ko-fi** | 开源项目常见的支持方式 | ✅ 用户自愿赞助 |

**阶段 3：长期变现（12 月+）**

| 方式 | 说明 | 预估价值 |
|------|------|---------|
| **Tax Engine API (SaaS)** | 88 国税制引擎作为独立 API 出售给 HR SaaS / EOR 公司 | $500-2000/年/客户 |
| **白标报告** | 给搬迁公司/咨询公司提供定制城市对比报告 | $50-200/份 |
| **数据集许可** | 反过来——当数据体系成熟后，将 WhichCity 的清洁数据集以 CC BY 许可出售 | 长期资产 |

### 10.3 投资者补充意见

Tax Engine API 是最有价值的长期变现路径。原因：
1. 100% 自有 IP，不依赖任何第三方
2. 88 国税制 + 6 种外派方案在市场上独一无二
3. 目标客户（EOR 公司如 Deel/Remote/Oyster）年收入已达数亿美元
4. 一个人维护的 API 完全可以支持小规模 B2B 销售

**但这是 12 月后的事。现在的任务是把产品做好、数据做干净。**
