# 审计工作分配表

## 团队成员与分工

| 审计员 | 负责领域 | 产出物 |
|--------|---------|--------|
| A 员（视觉设计） | 配色、对比度、字体、阴影、圆角、"AI味" | 色值统计 + WCAG 计算 + 设计语言审查 |
| B 员（布局响应式） | 断点、网格、移动端、固定尺寸、SSR 闪烁 | 断点映射表 + 移动端不可用区标注 |
| C 员（交互/UX） | 事件绑定、状态管理、URL sync、加载状态、恢复路径 | 交互流程图 + 关键路径失败点列表 |
| D 员（国际化/内容） | i18n 覆盖、硬编码字符串、CJK 字体、数据源链接 | i18n 覆盖率矩阵 + 硬编码扫描报告 |
| E 员（冗余/代码质量） | 行数违规、样式重复、功能膨胀、认知负荷 | 代码统计 + 功能价值评估矩阵 |

## 每人执行的三重验证方式

所有审计员必须以至少三种独立方法验证各自的结论：

### A 员
1. **代码精读**：逐行阅读 globals.css, tailwind.config.ts, 全部 8 个组件的样式代码
2. **grep 统计**：量化色值使用频率、darkMode 三元数量、圆角/阴影/字号分布
3. **WCAG 手动计算**：对所有 text-on-background 组合计算对比度比值

### B 员
1. **JSX 结构分析**：提取所有 grid/flex/min-w/max-w/breakpoint class
2. **断点逻辑走读**：追踪 1080px/420px/768px 等断点在代码中的使用
3. **固定尺寸检测**：搜索 TEMP_H/RAIN_H/h-7/w-7 等硬编码尺寸

### C 员
1. **事件绑定审查**：检查 onClick/onKeyDown/onChange 的覆盖度和一致性
2. **状态管理审查**：分析 useSettings hook + localStorage/sessionStorage 使用模式
3. **恢复路径分析**：从 error.tsx/not-found.tsx 出发分析用户能否恢复正常使用

### D 员
1. **i18n key 频率统计**：grep 所有 t("...") 调用并与 i18n.ts 定义交叉
2. **硬编码英文扫描**：grep 所有 app/ 和 components/ 中的英文字面量
3. **CJK 字体审查**：检查 layout.tsx + tailwind.config.ts 的字体加载配置

### E 员
1. **行数统计**：wc -l 检测所有组件是否超过 300 行规则
2. **样式重复统计**：量化 darkMode 三元、CardBg、selectCls 等重复模式  
3. **功能复杂度评估**：评估每个功能模块的 Tab/按钮/选项数量 vs 用户价值
