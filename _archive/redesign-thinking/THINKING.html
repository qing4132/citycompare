WhichCity — 从零开始的产品思考

不是"改什么"，而是"到底要做什么"。

一、用户到底在想什么

抛掉所有现有页面。只想一个人。

他叫小张，28 岁，软件工程师，在北京。他周日晚上躺在床上刷手机，脑子里冒出一个念头：

"如果我不在北京待了，去哪里比较好？"

他打开 whichcity.run。从这一刻起，他的思维路径是：

  1. 我有一个模糊的方向吗？
    有 → 他心里已经有几个城市（东京？新加坡？）→ 他要评估
    没有 → 他不知道去哪 → 他要发现
  2. "评估"的核心问题是什么？
    不是"安全指数是 87.8"。是：
    — 去了之后 每月能存多少钱？
    — 安不安全？ 治安好不好、社会稳不稳
    — 日子舒不舒服？ 气候、语言、工时
  3. "发现"的核心问题是什么？
    不是"按安全指数从高到低排"。是：
    — 以我的职业和收入，哪些城市 存钱最多？
    — 哪些城市 又安全又便宜？
    — 具体到"我"，不是泛泛的排名

关键洞察

用户不是来"看数据"的。他是来"做决定"的。
每一个像素都应该帮他更快地做出"去"或"不去"的判断。
如果一块信息不能帮助决策，它就不应该出现在屏幕上。

二、产品到底需要什么

从小张的思维路径倒推，产品需要的能力是：

能力小张的话必要性
搜索一个城市"我想看看东京"🔴 没有这个产品不存在
评估一个城市"东京适不适合我"🔴 这是核心价值
发现好的城市"有什么城市值得去"🟡 重要，但可以很简单
对比两个城市"东京和新加坡哪个好"🟡 有用，但不是第一天需要
理解数据来源"这数据靠谱吗"⚪ 信任背书，可以很轻

现有五个页面的审判

页面判决理由
首页 不应该单独存在 现在的首页就是一个搜索框+热门城市。这不是一个"页面"，这是城市页的入口状态。
城市详情 核心，保留 产品的全部价值在这里。但需要重新设计。
排行榜 改造为"发现" 现在是数据库表格式排行，应该变成"帮我找到适合我的城市"的工具。
对比页 延后 有价值但不是核心路径。第一版可以不做，用城市页上的"相似城市"覆盖。
方法论 降为 Footer 内容 一个独立页面太重了。数据来源和计算方式可以折叠在 Footer，或用 tooltip 附在每个数据点旁边。

三、最小产品是什么

结论：WhichCity 只需要两个"视图"，甚至可以在一个页面上。

视图 A：搜索/发现（空状态）

用户还没选城市时看到的东西。

空状态 — 搜索/发现
WHICHCITY
如果可以去任何地方，
你会选哪座城市？
🔍 输入城市名…
或 按存钱率看排名

就这么多。不需要热门城市网格、不需要 12 个 pill、不需要统计数字。
一个问句 + 一个搜索框 + 一个排行入口链接。

视图 B：城市评估（有内容状态）

用户选了城市后看到的东西。这是产品的全部价值。后面详细展开。

发现（排行）在哪？

两种路径：

  • 路径 1：搜索框下方的"按存钱率看排名"链接，跳到一个简单列表页。不是 16 个 tab 的数据库表格——而是一个有明确观点的排行："如果你是软件工程师，存钱最多的城市是这些。"
  • 路径 2：城市页底部的"相似城市"卡片，引导横向探索。

对比在哪？

第一版不做独立对比页。对比需求通过两种方式覆盖：

  • 城市页上的"相似城市"区块已经给出差异数据
  • 未来如果需要，可以做成城市页的"分屏模式"而不是独立页面

方法论在哪？

每个数据区块的 Footer 注明来源。展开详情面板里有计算说明。不需要独立页面。

四、站点结构

whichcity.run ├── /[locale]/ ← 搜索/发现（空状态） ├── /[locale]/city/[slug] ← 城市评估（核心） └── /[locale]/ranking ← 排行列表（发现入口，极简） 共 3 个路由。不是 5 个。 对比页和方法论页从第一版中移除。

URL 保持兼容：

  • /zh/compare/* → 301 到首页（或做简化版）
  • /zh/methodology → 301 到首页带锚点

五、城市页的重新设计

这次不从"区块"开始，从用户的阅读节奏开始。

阅读节奏

用户打开东京的页面。他的眼睛会按这个顺序扫描：

时间眼睛在哪脑子在想页面给什么
0-1 秒 最大的字 "对，是东京" 🇯🇵 东京 · 日本
1-3 秒 颜色最强的区域 "这地方怎么样？" 一张有颜色的卡片：
"安全 A · 月存 ¥8 万 · 医疗前 15%"
3-10 秒 卡片下方 "具体数字是？" 三个大数字：¥33万 / ¥25万 / ¥8万
10-30 秒 继续往下滚 "其他方面呢？" 安全横条 + 生活画像网格
30 秒+ 感兴趣才看 "税怎么算的？子指标？" 折叠区
离开前 底部 "还有别的选择吗？" 相似城市卡片 → 横向探索

信息层级

核心原则：每下一屏，信息密度升高，情感浓度降低。
首屏 = 结论（情感驱动，颜色、大字）
中段 = 证据（数字、图表）
底部 = 细节（折叠区、来源标注）

始终可见（不折叠）的信息：

  • 城市名 + 国旗 + 国家 + 一句话简介
  • 结论卡片（评级 + 关键数字 + 标签）
  • 收入 / 支出 / 结余 三个数字
  • 安全评级 + 四根横条
  • 生活网格（6-8 个事实）
  • 相似城市

折叠的信息：

  • 税务明细（累进税率拆分）
  • 安全/医疗/治理子指标
  • 月度气候图
  • 数字游民信息

不展示的信息：

  • 基尼系数、性别工资差距、高等教育率、净迁移 — 太学术
  • 监管质量、宽带普及率 — 代理指标，不直观
  • 所有 confidence 值 — 仅在低置信时出⚠️

结论卡片的再思考

上一版的结论卡片是自然语言段落。但用户扫描速度极快，段落需要"阅读"。

新想法：结论卡片不写段落，用结构化的 key-value：

结论卡片 · 方案 B — 结构化
✦ WhichCity 评估 A
月存 ¥8 万
安全 88/100
医疗 前 15%
AQI 42 优
极低犯罪 中等税负 英语弱

对比方案 A（段落式）：

结论卡片 · 方案 A — 段落
✦ WhichCity 评估
安全评级 A，治理与医疗均为全球前列。 软件工程师税后月入约 ¥33万， 扣除生活成本后月结余约 ¥8万。
极低犯罪 中等税负 英语弱

我倾向方案 B：扫描速度更快，关键数字一眼看完，不需要"阅读"。但这个由你决定。

六、排行页的重新设计

现有排行榜：16 个 tab、气候筛选、子指标排序 — 这是给数据分析师用的，不是给小张用的。

小张想知道的是："以我的情况，去哪最划算？"

极简排行

只做一个排行：按你选的职业的月结余排序。这是最直觉的"划算"定义。

排行页 · 极简
软件工程师，去哪存钱最多？
基于税后收入减生活成本 · 换职业
#城市月结余安全
1🇦🇪 迪拜$5,10073
2🇸🇬 新加坡$4,06091
3🇨🇭 苏黎世$3,80089
4🇺🇸 旧金山$3,50062
5🇺🇸 西雅图$3,40065
… 点击任意行 → 城市详情

辅助列不超过 2 个。默认只显示安全指数（因为"又赚钱又安全"是最常见的筛选心理）。

如果将来需要更多维度（"最适合家庭"、"最佳气候"），可以用"视角"概念：

  • 💰 存钱最多（默认）
  • 🛡️ 最安全
  • ☀️ 最佳气候
  • ⚖️ 综合平衡

但第一版只做"存钱最多"。一个排序做到极致。

七、不做什么

明确说"不做"比"做什么"更重要。

第一版不做的事情：
  • 对比页 — 用"相似城市"替代
  • 独立方法论页 — 折叠在数据来源里
  • 16 个排行维度 — 只做月结余排行
  • 气候筛选 — 城市页上有气候信息就够了
  • 子指标排序 — 过度专业
  • 分享面板（8 个社交平台） — 一个"复制链接"就够
  • 设置面板里的消费档位 — 默认适度消费，暂不提供选项
  • 收入模式切换（gross/net/expat） — 默认税后，暂不提供选项

这些功能不是没价值，是此刻不是最重要的。等核心体验做到极致，再一个一个加回来。

八、技术架构

文件结构

redesign/ ├── THINKING.html ← 这个文件（设计思考） ├── city/ │ └── CityPage.tsx ← 城市页主组件 ├── home/ │ └── HomePage.tsx ← 首页（搜索 + 入口） ├── ranking/ │ └── RankingPage.tsx ← 排行（极简列表） ├── shared/ │ ├── Nav.tsx ← 导航栏（< 50 行） │ ├── Footer.tsx ← 页脚 │ └── theme.css ← CSS 变量（暗色模式的唯一来源） └── lib/ ├── grading.ts ← 评级计算（S/A/B/C/D） ├── insights.ts ← insight 文案生成 └── format.ts ← 数字格式化

暗色模式策略

一个 CSS 文件定义所有颜色变量。.dark class 覆盖。
组件里零 darkMode 三元表达式。颜色全部用 text-[--fg] 或 CSS 变量。

数据流

组件不做计算。所有计算在 redesign/lib/ 的纯函数里完成。
组件只接收计算好的数据，只做渲染。

数据流：
page.tsx (服务端) → 加载 City + 计算 → 传给 CityPage.tsx (客户端) → 纯渲染
