# 自动更新设计方案

> 状态: **设计阶段**（暂缓实现）
> 目标: 所有数据源都能通过 GitHub Actions 在各自合理周期内自动更新

## 分层更新策略

不需要每次全量刷新所有 150 城市的所有字段。按数据源更新频率和复杂度分 4 层:

### Layer 0 — 已自动化 (现状)

| 数据 | 频率 | Workflow |
|------|------|---------|
| exchange-rates.json | 每日 | `update-exchange-rates.yml` |

### Layer 1 — 低成本自动化 (公开 API，无限制)

| 数据 | 频率 | API | 估计开发 |
|------|------|-----|---------|
| doctorsPerThousand | 年度 | World Bank API | 2h |
| hospitalBedsPerThousand | 年度 | World Bank API | — (同上脚本) |
| uhcCoverageIndex | 双年 | World Bank API | — |
| lifeExpectancy | 年度 | World Bank API | — |
| outOfPocketPct | 年度 | World Bank API | — |
| bigMacPrice | 半年 | GitHub CSV | 1h |
| gpiScore | 年度 | Vision of Humanity | 1h |

**实现方式**: 单个 GitHub Actions workflow，cron `0 6 1 1,7 *` (每年 1 月和 7 月 1 日)。
脚本: 扩展现有 `collect-new-indicators.py` → 写入 `data/cities-source.json` → 运行 `export.mjs` → 自动 PR。

```yaml
# .github/workflows/update-world-bank.yml (草案)
name: Update World Bank Indicators
on:
  schedule:
    - cron: "0 6 1 1,7 *"  # Jan 1 + Jul 1
  workflow_dispatch:

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-node@v5
        with: { node-version: 22 }
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: python scripts/collect-new-indicators.py --target data/cities-source.json
      - run: node data/scripts/export.mjs
      - run: node data/scripts/validate.mjs
      - uses: peter-evans/create-pull-request@v6
        with:
          title: "data: auto-update World Bank indicators"
          branch: auto/world-bank-update
          commit-message: "data: auto-update World Bank indicators"
```

### Layer 3 — 高复杂度 (反爬/付费)

| 数据 | 频率 | 来源 | 难度 | 限制 |
|------|------|------|------|------|
| numbeoSafetyIndex | 半年 | Numbeo | 高 | 限频 + IP 封禁 |
| costModerate | 半年 | Numbeo | 高 | 同上 |
| monthlyRent | 半年 | Numbeo | 高 | 同上 |
| housePrice | 半年 | Numbeo + GPG | 高 | 同上 |
| professions | 年度 | 多来源 | 极高 | 需人工交叉验证 |
| averageIncome | 年度 | 多来源 | 极高 | 同上 |
| gallupLawOrder | 年度 | Gallup | — | 付费报告 |
| democracyIndex | 年度 | EIU | — | 付费报告 |

**Numbeo 策略**:
- 使用 `verify-numbeo-data.mjs`（已实现 Clash 代理节点切换 + 断点续传）
- GitHub Actions 中通过 self-hosted runner（家庭网络 + 代理）运行
- 每次只更新 30 个城市（分批，每月一批，5 个月覆盖全部）
- 失败自动跳过，下次继续

**渐进式更新规则**:
```
月份 1: 城市 1-30 的 Numbeo 数据
月份 2: 城市 31-60
月份 3: 城市 61-90
月份 4: 城市 91-120
月份 5: 城市 121-150
月份 6: 全量验证 + 对比报告
```

## GitHub Actions Workflow 设计

### 统一结构

```yaml
# 通用模板
name: Update {Source}
on:
  schedule:
    - cron: "{schedule}"
  workflow_dispatch:

permissions:
  contents: write
  pull-requests: write

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-node@v5
        with: { node-version: 22 }

      # 1. 获取数据
      - name: Fetch data
        run: node scripts/fetch-{source}.mjs --target data/cities-source.json

      # 2. 导出 + 验证
      - name: Export & validate
        run: |
          node data/scripts/export.mjs
          node data/scripts/validate.mjs

      # 3. 创建 PR (不自动合并)
      - uses: peter-evans/create-pull-request@v6
        with:
          title: "data: auto-update {source} [{date}]"
          branch: auto/{source}-update
          body: |
            Auto-generated by {workflow}.
            Review the diff before merging.
```

### 失败处理和告警

- 每个 workflow 使用 `continue-on-error: false`
- 导出/验证失败 → workflow 失败 → GitHub 通知
- 数据获取失败（网络/API 变更）→ 记录到 `data/changelog.json` → 告警
- 数据源变更检测: 比较获取的数据结构与上次快照，字段消失/新增 → 告警

### 数据源变更检测

```javascript
// 伪代码: 每个 fetch 脚本末尾
const oldSchema = loadSchema("data/sources/{source}-schema.json");
const newSchema = extractSchema(fetchedData);
if (!isCompatible(oldSchema, newSchema)) {
  console.error("⚠ Data source schema changed!");
  // 不写入数据，只发告警
  process.exit(2); // 特殊退出码
}
```
